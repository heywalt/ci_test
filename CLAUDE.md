## Development Practices

- TDD using the red-green-refactor pattern
- Run `mix format` before committing code changes to Git

### Code Style

- Use consistent indentation (spaces, not tabs)
- Never use nested alias syntax (e.g. `alias CQRS.Enrichments.Commands.{RequestProviderEnrichment, CompleteProviderEnrichment}`)
- Use individual alias statements for better readability
- Never use single line function syntax (`def foo(x), do: x + 1`) if the function runs onto multiple lines
- Always convert single-line syntax function definitions that wrap to multiple lines into multi-line function definition syntax
- Prefer `List.first/1` over `hd/1` because it's safer for empty lists (returns `nil` instead of raising)

### Git Practices

- Keep commit messages concise but descriptive
- Do not mention adding tests in commit messages
- Never include "Generated by" or "Co-authored by" lines in commit messages
- Do not include "Generated by" messaging in PR descriptions
- Keep PR titles and descriptions concise

## Project Overview

Walt UI is a CRM-like system built with Elixir/Phoenix using CQRS/Event Sourcing architecture. It manages contacts, enriches them with external data, handles email/calendar integration, and provides search capabilities.

### Core Features
- **Contact Management**: CRUD operations, bulk imports, Move Score (Priority Touch Time) scoring, favorites
- **Data Enrichment**: Automated enrichment via Endato, Faraday, Gravatar, and AI-powered scoring (Jitter)
- **Email/Calendar Sync**: Google Gmail and Calendar integration for communication tracking
- **Search**: TypeSense-powered full-text search with real-time updates
- **Task Management**: Contact-linked tasks with priorities, due dates, and reminders
- **Billing**: Subscription management via Stripe and RevenueCat

## Tech Stack

- **Backend**: Elixir 1.16.1, Phoenix 1.7.19, Commanded (CQRS/ES)
- **Frontend**: Phoenix LiveView, Alpine.js 3.14, Tailwind CSS 4.0
- **Database**: PostgreSQL 14.13 with Event Store
- **Search**: TypeSense
- **Background Jobs**: Oban, Broadway (SQS/Pub-Sub)
- **Authentication**: Auth0, Google OAuth, Apple Sign-in
- **Payments**: Stripe, RevenueCat
- **Infrastructure**: Docker, GCP, Terraform

## Architecture Patterns

### CQRS/Event Sourcing
- Commands are handled by aggregates (Leads, Enrichments, Meetings) in `/lib/walt_ui/`
- Events stored in PostgreSQL-based Event Store
- Events are projected to read models in `/lib/walt_ui/projections/`
- Process managers coordinate complex workflows (e.g., contact unification, enrichment)
- Event handlers implement side effects (e.g., triggering background jobs)

#### Important: New Event Handler Configuration
**ALWAYS** remind me to add `start_from: :current` when creating new event handlers to prevent processing historical events:
```elixir
use Commanded.Event.Handler,
  application: CQRS,
  name: __MODULE__,
  start_from: :current  # <- CRITICAL: Prevents processing all historical events
```
Without this, new handlers will process ALL events from the beginning of time, potentially triggering thousands of unnecessary operations.

## Testing Approach

- TDD with red-green-refactor pattern
- Uses ExUnit for testing
- Mox for mocking external dependencies
- Tests are located in `/test/` directories within each app
- `assert_async` utilities for testing asynchronous operations
- Factory pattern with ExMachina for test data
- Tests that hit the database run inside a transaction that is rolled back after the test completes. This provides isolation even when running tests in parallel and ensures automatic cleanup of database state. There is no need to drop or reset the database before running tests.
